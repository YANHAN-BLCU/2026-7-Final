## 一、倒排索引 & 布尔检索

### 1. 倒排索引构建

两步走：
- **词条化（Tokenization）**：文档 → 切词 → 归一化（大写转小写、词干提取等）
- **排序+归并**：`(词项, docID)` 对按词项排序，相同词项合并为倒排记录表（postings list）

```
原文档: Doc1="I like cats", Doc2="I like dogs"
(like,1), (like,2), (cats,1), (dogs,2), (I,1), (I,2)
→ 排序后:
cats→[1], dogs→[2], I→[1,2], like→[1,2]
```

### 2. 合并两个倒排记录表的时间复杂度

**O(x + y)** — x和y是两个表的长度。双指针扫描，谁小移谁，线性时间。

### 3. 布尔查询优化

多个AND条件时，**优先合并最短的倒排记录表**（最短优先策略），减少中间结果集大小。

```
查询: cat AND dog AND mouse
表长: cat=100, dog=200, mouse=500
最优: (cat ∩ dog) ∩ mouse → 先交最短的两个
```

### 4. 基于MapReduce的索引构建

- **Map阶段**：每个文档分给一个Mapper，输出 `(词项, docID)` 键值对
- **Shuffle阶段**：按词项分组，同一词项分到同一Reducer
- **Reduce阶段**：对每个词项，收集所有docID，排序后写倒排记录表

## 二、索引压缩 & TF-IDF

### 1. 为什么要压缩

- 倒排索引可能比原文还大（词项表 + 指针 + docID）
- **节省磁盘I/O**：压缩后读更快
- **更多数据进内存**：减少磁盘访问
- 压缩解压速度比磁盘I/O快得多

### 2. 词典压缩 — 单一字符串存储

把所有词项连续写成一个长字符串，每个词项用 `(偏移量, 长度)` 定位：

```
偏移量: 0  3  7  12  18
字符串: cat\0dog\0fish\0bird\0...
```

省掉了每个词项独立存储时的指针开销（每个指针8字节 → 每词项省~60%空间）

### 3. 单一字符串方式下按块存储

把词典分块，每块内部连续存字符串，块边界存指针。查词时先二分块，再在块内顺序扫描。

### 4. 前端编码（Front Coding）

利用相邻词项共享前缀。适用于排序后的词典：

```
原: cat, category, caterpillar, dog, dogmatic
编码: cat, 5egory, 8pillar, dog, 4matic
```

共享的前缀用数字表示长度，差量编码。

### 5. 倒排记录表压缩 — 对docID间隔编码

不存绝对docID，存与上一条的差值（gap）：

```
原: [1, 5, 8, 20, 25] → gap: [1, 4, 3, 12, 5]
```

差值远小于原值，用变长编码（如γ编码、VB编码）压缩。

### 6. Jaccard系数

$$J(A,B) = \frac{|A \cap B|}{|A \cup B|}$$

衡量两个集合的重合度，值域[0,1]。常用于文档相似度、去重。

### 7. 对数词频

$$weight(t,d) = \begin{cases} 1 + \log_{10} tf(t,d) & tf > 0 \\ 0 & tf = 0 \end{cases}$$

词频取对数，防止高频词过度放大。

### 8. IDF（逆文档频率）

$$idf(t) = \log \frac{N}{df(t)}$$

N=文档总数，df(t)=包含词项t的文档数。罕见词IDF高，常见词IDF低（"的""了"被压下去）。

### 9. TF-IDF权重

$$tfidf(t,d) = (1 + \log tf(t,d)) \times \log \frac{N}{df(t)}$$

词在文档中出现越多（TF高）→ 越重要；词在全部文档中出现越少（IDF高）→ 越有区分度。

## 三、评价

### 1. IR中评价什么

| 维度 | 含义 |
|:----|:-----|
| **相关性** | 返回结果和查询需求是否匹配 |
| **覆盖率** | 检索出了多少相关文档 |
| **效率** | 检索速度、索引大小 |
| **用户满意度** | 点击率、停留时间等 |

### 2. 召回率（Recall）

$$Recall = \frac{|\text{相关且被检索到}|}{|\text{全部相关文档}|}$$

"该捞的捞上来多少"

### 3. 正确率（Precision）

$$Precision = \frac{|\text{相关且被检索到}|}{|\text{全部检索结果}|}$$

"捞上来的有多少是好的"

**两者通常 trade-off** — 提高召回一般会降低准确率，反之亦然。

### 4. 搜索结果摘要的基本类型

| 类型 | 说明 |
|:----|:-----|
| **静态摘要** | 固定取文档前N个字（简单但可能不相关） |
| **查询相关摘要** | 取查询词在文档中出现的上下文片段（效果好，标准做法） |
| **结构化摘要** | 包含标题、URL、发布时间等元信息（搜索引擎结果页） |

### 5. 提高召回率的方法

- 查询扩展（同义词、相关词）
- 相关反馈（用户标"这个相关" → 调整权重）
- 全局分析（词项聚类、潜在语义分析）
- 降低相关性阈值

## 四、广告 & 拍卖

### 1. 上下文相似度

将广告与页面内容匹配——广告内容和网页主题越相关，越可能投放。用TF-IDF/向量空间模型计算。

### 2. 传统广告不足

- 难以定向（不知道谁在看）
- 浪费曝光（不相关的人看到也没用）
- 效果难衡量
- "一半广告费被浪费了，但不知道是哪一半"

### 3. 互联网广告的主要形式

| 形式 | 说明 |
|:----|:-----|
| **展示广告** | 横幅、插屏、视频贴片 |
| **搜索广告** | 搜索关键词触发，按点击付费（CPC） |
| **社交广告** | 基于用户画像定向投放 |
| **原生广告** | 伪装成内容的广告 |
| **程序化广告** | 实时竞价（RTB）自动购买 |

### 4. 次高价拍卖（Second-Price Auction / GSP）

搜索引擎广告位拍卖的核心机制：

- 广告主对关键词出价（bid）
- **出价最高的赢得广告位，但只需付第二高的出价**
- 优点：鼓励广告主出真实估价（truthful bidding），避免竞价博弈

**相关考法：**
- 问答题：解释次高价拍卖为什么鼓励真实出价
- 编程题：实现次高价拍卖函数——输入出价列表，输出中标者及其支付价格

```
输入: [A:3元, B:2元, C:1元]
输出: 中标=A, 支付=2元
```

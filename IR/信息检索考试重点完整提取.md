## 1. 倒排索引与布尔检索

### 1.1 倒排索引的构建

来源：`lecture1-booleanretrieval.pdf` pp.26-32；`lecture4-indexconstruction.pdf` pp.20-26

倒排索引（inverted index）的核心思想：对每个词项 `t`，保存所有包含该词项的文档列表。每篇文档用唯一的 `docID` 表示，通常是正整数。

基本结构：

```text
词典 Dictionary:
  term -> df, pointer

倒排记录表 Postings list:
  term -> docID list
```

其中：

- `term`：词项。
- `df`：document frequency，包含该词项的文档数。
- `pointer`：指向该词项倒排记录表的位置。
- postings list：按照 `docID` 从小到大排序的文档编号列表。

构建流程：

1. 文档输入。
2. 词条化（tokenization），把文档切成词条流。
3. 语言分析与归一化，例如大小写统一、词干还原、词形归并等。
4. 生成 `<term, docID>` 二元组。
5. 按 `term` 排序；同一个 `term` 下再按 `docID` 排序。
6. 合并同一篇文档中重复出现的词项。
7. 拆成词典和倒排记录表两部分，并记录每个词项的 `df`。

示意：

```text
Doc 1: I did enact Julius Caesar
Doc 2: So let it be with Caesar

生成二元组:
<i,1>, <did,1>, <caesar,1>, <so,2>, <caesar,2>

排序合并:
caesar -> [1,2]
did    -> [1]
i      -> [1]
so     -> [2]
```

为什么倒排记录表按 `docID` 排序：

- 便于快速求交集、并集和差集。
- 布尔查询中的合并可以线性完成。
- 便于后续使用 docID 间隔压缩。

### 1.2 合并两个倒排记录表的时间复杂度

来源：`lecture1-booleanretrieval.pdf` pp.35-37

以 `Brutus AND Caesar` 为例：

1. 在词典中定位 `Brutus`，取出对应 postings list。
2. 在词典中定位 `Caesar`，取出对应 postings list。
3. 合并两个 postings list，即求交集。

合并方法：双指针扫描。

```text
p1 指向表 A 当前元素
p2 指向表 B 当前元素

若 A[p1] == B[p2]：输出该 docID，两个指针都后移
若 A[p1] < B[p2]：p1 后移
若 A[p1] > B[p2]：p2 后移
```

若两个倒排记录表长度分别为 `x` 和 `y`，则复杂度为：

```text
O(x + y)
```

原因：两个指针都只从前到后移动，每个倒排记录最多被访问一次。

### 1.3 布尔查询优化

来源：`lecture1-booleanretrieval.pdf` pp.38-41

布尔查询包括：

- `AND`：求交集。
- `OR`：求并集。
- `NOT`：求差集，例如 `Brutus AND NOT Caesar`。
- 一般布尔表达式，例如 `(Brutus OR Caesar) AND NOT (Antony OR Cleopatra)`。

多个 `AND` 条件时，处理顺序会影响效率。

优化原则：

```text
优先合并最短的倒排记录表，即 df 最小的词项先处理。
```

例：

```text
Brutus     -> 7 个 docID
Caesar     -> 8 个 docID
Calpurnia  -> 2 个 docID

查询:
Brutus AND Caesar AND Calpurnia

优化处理:
(Calpurnia AND Brutus) AND Caesar
```

这样可以尽早得到较小的中间结果，减少后续合并成本。

更一般的优化策略：

对于 `(madding OR crowd) AND (ignoble OR strife)` 这类查询：

1. 获取每个词项的 `df`。
2. 对每个 `OR` 子表达式，用其中词项 `df` 之和保守估计结果规模。
3. 按估计规模从小到大处理各个子表达式。

### 1.4 基于 MapReduce 的索引构建

来源：`lecture4-indexconstruction.pdf` pp.28-36

大规模 Web 数据不能靠单机完成索引构建，必须使用分布式计算集群。MapReduce 是一种鲁棒、简单的分布式计算框架，Google 的索引构建系统中多个步骤都采用 MapReduce 实现。

分布式索引构建基本角色：

- `Master`：主节点，负责任务分配和调度。
- `Parser`：分析器，读取文档并输出 `(term, docID)`。
- `Inverter`：倒排器，收集某一词项分区的所有 `(term, docID)`，排序并写入倒排记录表。

流程：

1. 将输入文档集切成多个 split，每个 split 是文档子集。
2. Master 将 split 分配给空闲 Parser。
3. Parser 逐篇读取文档，输出 `(term, docID)` 对。
4. Parser 按词项分区，例如 `a-f`, `g-p`, `q-z`。
5. Inverter 收集对应分区的所有 `(term, docID)`。
6. Inverter 排序并生成倒排记录表。

MapReduce 表达：

```text
Map:
  输入文档 d
  输出 <term, docID>

Shuffle:
  按 term 分组，把相同 term 的记录送到同一 Reduce

Reduce:
  输入 <term, list(docID)>
  输出 <term, postings list>
```

课件例子：

```text
Map:
d2: C died
  -> <C,d2>, <died,d2>
d1: C came, C c'ed
  -> <C,d1>, <came,d1>, <C,d1>, <c'ed,d1>

Reduce:
<C, (d2,d1,d1)> -> <C, (d1:2, d2:1)>
<died, (d2)>    -> <died, (d2:1)>
<came, (d1)>    -> <came, (d1:1)>
```

## 2. 索引压缩

### 2.1 为什么要压缩

来源：`lecture5-indexcompression(1).pdf` pp.9-12

一般意义上的压缩：用短编码串代替长编码串。

IR 中压缩的动机：

- 减少磁盘空间，节省存储开销。
- 增加内存中可存储的索引内容，从而加快检索速度。
- 加快从磁盘到内存的数据传输速度。
- 在解压速度足够快的前提下，“读压缩数据到内存 + 内存中解压”通常比直接读未压缩数据更快。

IR 中主要压缩对象：

1. 词典压缩：目标是尽量让词典放入内存。
2. 倒排记录表压缩：目标是减少磁盘空间和读取时间。倒排记录表通常远大于词典，至少大 10 倍以上。

压缩类型：

- 有损压缩：丢弃一些信息，例如统一小写、去除停用词、词干还原、去掉数字。
- 无损压缩：保留所有信息。索引压缩通常使用无损压缩。

### 2.2 词典压缩：单一字符串存储

来源：`lecture5-indexcompression(1).pdf` pp.18-21

定长数组存储词典的缺点：

- 对每个词项分配固定长度，短词会浪费空间。
- 不能处理超过固定长度的长词。
- 英语词项平均长度约为 8 字节，但定长方式可能给每个词项分配 20 字节。

单一字符串方式：把整部词典看成一个连续字符串。

每个词项只需保存：

- 文档频率 `df`：4 字节。
- 指向倒排记录表的指针：4 字节。
- 指向词典字符串中该词项起始位置的指针：约 3 字节。
- 词项字符串本身：平均约 8 字节。

Reuters RCV1 课件估算：

```text
定长方式:
(20 + 4 + 4) * 400,000 = 11.2 MB

单一字符串方式:
400,000 * (4 + 4 + 3 + 8) = 7.6 MB
```

优点：明显降低词典空间开销。

### 2.3 词典压缩：单一字符串方式下按块存储

来源：`lecture5-indexcompression(1).pdf` pp.22-26

进一步压缩思路：不是给每个词项都保存一个字符串指针，而是按块保存。

例如每 4 个词项一块：

- 每块只保留 1 个指向字符串起始位置的指针。
- 块内每个词项保存长度。
- 查询时先二分定位到块，再在块内顺序扫描。

课件空间估算：

```text
不按块:
每 4 个词项需要 4 * 3 = 12B 指针

按块:
每 4 个词项需要 3B 块地址 + 4B 长度 = 7B

每 4 个词项节省:
12B - 7B = 5B

词典空间:
从 7.6MB 压缩到约 7.1MB
```

代价：查询稍慢。因为先二分查找块，再在块内线性查找。课件例子中块大小为 4 时，平均查找步数从约 1.6 步增加到 2 步。

### 2.4 词典压缩：前端编码

来源：`lecture5-indexcompression(1).pdf` p.27

前端编码（front coding）利用排序后相邻词项常有公共前缀这一特点。

例：

```text
automata
automate
automatic
automation
```

它们共享前缀 `automat`，可以编码为：

```text
automat* a
         e
         ic
         ion
```

课件表示中，用 `*` 标记公共前缀结束，用特殊符号表示后续词项继承此前缀。

适用场景：

- 词典已经按字典序排序。
- 相邻词项具有较长公共前缀。

优点：进一步压缩词典字符串。

### 2.5 倒排记录表压缩：对 docID 间隔编码

来源：`lecture5-indexcompression(1).pdf` pp.30-33

倒排记录表压缩的关键：不直接存储 docID，而是存储相邻 docID 的间隔（gap）。

因为每个倒排记录表中的 docID 按从小到大排序，所以可以做差分。

例：

```text
原 docID:
[283154, 283159, 283202]

间隔编码:
[283154, 5, 43]
```

另一个简单例子：

```text
原 docID:
[1, 5, 8, 20, 25]

gap:
[1, 4, 3, 12, 5]
```

为什么有效：

- 高频词项的 postings list 很长，docID 间隔通常较小。
- 小整数可以用更短编码表示。
- 可变长编码对小 gap 用短编码，对大 gap 用长编码。

目标：

- 对罕见词项，较大的 gap 仍可使用较多比特。
- 对高频词项，小 gap 只用很少比特。
- 压缩后每个 docID 使用的平均位数远小于普通定长整数。

## 3. Jaccard、对数词频、IDF 与 TF-IDF

### 3.1 Jaccard 系数

来源：`lecture6-tfidf.pdf` pp.15-17

Jaccard 系数是计算两个集合重合度的常用方法。

公式：

```text
Jaccard(A, B) = |A ∩ B| / |A ∪ B|
```

性质：

- `Jaccard(A, A) = 1`
- 如果 `A ∩ B` 为空，则 `Jaccard(A, B) = 0`
- 取值范围 `[0, 1]`
- 两个集合不需要同样大小

课件例子：

```text
查询 q: "ides of March"
文档 d: "Caesar died in March"

A = {ides, of, March}
B = {Caesar, died, in, March}

A ∩ B = {March}，大小为 1
A ∪ B = {ides, of, March, Caesar, died, in}，大小为 6

Jaccard(q, d) = 1 / 6
```

不足：

- 不考虑词项频率，即词项在文档中出现多少次。
- 不考虑罕见词比高频词信息量更大。
- 没有充分处理文档长度因素。

### 3.2 对数词频

来源：`lecture6-tfidf.pdf` pp.23-24

词项频率 `tf(t,d)`：词项 `t` 在文档 `d` 中出现的次数。

原始 tf 的问题：

- 如果某词在 A 文档出现 10 次，在 B 文档出现 1 次，A 通常更相关。
- 但相关度不应简单认为 A 是 B 的 10 倍。
- 相关度不会与 tf 成正比增长。

对数词频权重：

```text
若 tf(t,d) > 0:
  w(t,d) = 1 + log10(tf(t,d))

若 tf(t,d) = 0:
  w(t,d) = 0
```

例：

```text
tf = 0    -> w = 0
tf = 1    -> w = 1
tf = 2    -> w ≈ 1.3
tf = 10   -> w = 2
tf = 1000 -> w = 4
```

查询-文档得分可用共同词项的对数词频之和：

```text
score(q,d) = sum_{t in q ∩ d} (1 + log10 tf(t,d))
```

如果查询和文档没有公共词项，则得分为 0。

### 3.3 IDF

来源：`lecture6-tfidf.pdf` pp.27-34

IDF 的动机：罕见词项比常见词项携带更多信息。

- 罕见词如 `ARACHNOCENTRIC`，如果出现在文档中，文档很可能与查询相关。
- 常见词如 `GOOD`, `INCREASE`, `LINE`，虽然包含它们的文档比不包含的更相关，但它们区分能力较弱。

文档频率：

```text
df(t) = 包含词项 t 的文档数
```

IDF：

```text
idf(t) = log10(N / df(t))
```

其中 `N` 是文档集中文档总数。

含义：

- `df(t)` 越大，`idf(t)` 越小。
- `df(t)` 越小，`idf(t)` 越大。
- IDF 是词项信息量的指标。
- 实际中使用 `log(N / df)` 而不是直接使用 `N / df`，是为了抑制 IDF 的影响。

课件样例中，当 `N = 1,000,000`：

```text
calpurnia  df=1         idf=6
animal     df=100       idf=4
sunday     df=1,000     idf=3
fly        df=10,000    idf=2
under      df=100,000   idf=1
the        df=1,000,000 idf=0
```

注意：

- IDF 会影响至少包含两个词项的查询的排序。
- 单词项查询中，IDF 对文档间排序基本没有影响，因为所有匹配文档都包含同一个查询词。
- 文档频率 `df` 比文档集频率 `cf` 更适合作权重计算。

### 3.4 TF-IDF 权重计算

来源：`lecture6-tfidf.pdf` pp.35-36

TF-IDF 是信息检索中最经典的权重计算方法。

定义：

```text
tf-idf(t,d) = tf_weight(t,d) * idf(t)
```

采用对数词频时：

```text
tf-idf(t,d) = (1 + log10 tf(t,d)) * log10(N / df(t))
```

若 `tf(t,d) = 0`，则 `tf-idf(t,d) = 0`。

直观含义：

- 词项在当前文档中出现越多，`tf` 越高，权重越大。
- 词项在整个文档集中越罕见，`idf` 越高，权重越大。
- 常见词即使出现，也会因为 `idf` 低而权重较小。

注意：`tf-idf` 中的 `-` 是连接符，不是减号。其他写法包括 `tf.idf`、`tf x idf`。

## 4. 检索评价、摘要与提高召回率

### 4.1 IR 中评价什么

来源：`lecture8-evaluation(1).pdf` pp.9-13

评估 IR 的目的：比较不同技术优劣，分析不同因素对系统的影响，促进检索系统改进。

IR 系统目标：在较少消耗下，尽快、全面、准确地返回结果。

评价内容：

1. 效率（Efficiency）
   - 时间开销。
   - 空间开销。
   - 响应速度。

2. 效果（Effectiveness）
   - 返回的文档中有多少是相关文档。
   - 所有相关文档中返回了多少。
   - 相关文档是否排在靠前位置。

3. 其他指标
   - 覆盖率（Coverage）。
   - 访问量。
   - 数据更新速度。
   - 点击率、用户行为、A/B 测试等。

公平评测通常需要：

- 相同的文档集合。
- 相同的查询主题集合。
- 相同的评价指标。
- 比较不同检索系统的结果。

### 4.2 召回率 Recall

来源：`lecture8-evaluation(1).pdf` pp.17-19, pp.21-26；`lecture9-queryexpansion.pdf` p.3

召回率又称查全率，衡量所有相关文档中有多少被系统返回。

公式：

```text
Recall = 正确检索出的相关文档数 / 全部相关文档数
       = 正确检索出 / (正确检索出 + 未检索出)
```

取值范围：

```text
R in [0, 1]
```

例：

```text
某查询实际有 100 篇相关文档
系统返回 200 篇，其中 80 篇真正相关

Recall = 80 / 100 = 0.8
```

召回率高表示“该找出的相关文档找出了较多”。

召回率的难点：

- 大规模语料中，很难列举某个查询的全部相关文档。
- 因此召回率难以准确计算。
- TREC 常用 Pooling 方法：把多个系统 Top N 结果汇成文档池，人工标注其中相关文档，近似作为相关文档集合。

### 4.3 正确率 Precision

来源：`lecture8-evaluation(1).pdf` pp.17-19, pp.21-25；`lecture9-queryexpansion.pdf` p.3

正确率又称查准率，衡量返回结果中真正相关的比例。

公式：

```text
Precision = 正确检索出的相关文档数 / 检索出的全部文档数
```

取值范围：

```text
P in [0, 1]
```

例：

```text
某查询实际有 100 篇相关文档
系统返回 200 篇，其中 80 篇真正相关

Precision = 80 / 200 = 0.4
```

Precision 与 Recall 的关系：

- 返回很少但非常确定的结果：Precision 可能高，Recall 可能低。
- 返回全部文档：Recall = 1，但 Precision 通常很低。
- 不同应用对二者侧重不同。

例：

- 垃圾邮件过滤更重视 Precision，避免把正常邮件误判成垃圾邮件。
- 有些搜索场景用户希望结果全一点，更重视 Recall。
- 有些搜索场景用户希望前几条很准，更重视 Precision。

Precision 和 Recall 的不足：

- 召回率难以计算。
- 两个指标分别衡量系统不同方面，不容易单独判断哪个系统更好。
- 它们基于无序集合，未考虑排序位置。
- 若两个系统返回相同数量的相关文档，一个排在前 10，另一个排在最后 10，集合式 P/R 可能相同，但用户体验明显不同。

补充指标：

```text
F1 = 2PR / (P + R)
```

更一般的 `F_beta` 可调节 Precision 和 Recall 的相对重要性；`beta > 1` 更重视 Recall，`beta < 1` 更重视 Precision。

### 4.4 搜索结果摘要的基本类型

来源：`lecture8-evaluation(1).pdf` pp.49-56；`lecture9-queryexpansion.pdf` pp.4-5

搜索结果通常以列表形式呈现，也就是常说的 “10 blue links”。每篇文档的描述很关键，因为用户常常根据标题、URL、元数据和摘要判断是否点击。

摘要基本类型：

1. 静态摘要（static summary）
2. 动态摘要（dynamic summary）

静态摘要：

- 不随查询变化。
- 对同一篇文档，无论输入什么查询，摘要都一样。
- 最简单方法：返回文档前 50 个左右的单词。
- 更复杂方法：从文档中选择重要句子组成摘要。
- 也可使用 NLP 启发式或机器学习方法。
- 最复杂方法是生成式摘要，但对多数 IR 应用来说不够成熟。

动态摘要：

- 依赖当前查询。
- 试图解释“为什么这篇文档会被返回”。
- 给出一个或多个包含查询词项的窗口（snippet）。
- 出现完整查询短语的 snippet 优先。
- 查询词项在小窗口中密集出现的 snippet 优先。
- 最终把多个 snippet 显示为摘要。

动态摘要生成条件：

- 需要缓存文档内容。
- 通过位置索引知道查询词项在文档中的出现位置。
- 文档缓存版本可能过时。
- 对非常长的文档，可只缓存短前缀。

### 4.5 提高召回率的方法

来源：`lecture9-queryexpansion.pdf` pp.6, pp.8-13, pp.40-49

提高召回率的核心思想：返回更多相关文档，即使这些文档不一定包含原查询词。

例：

```text
查询 q: aircraft
某相关文档只包含 plane，不包含 aircraft
简单 IR 系统不会返回该文档
通过相关反馈或查询扩展，可以把 plane 关联进来
```

两类方法：

1. 局部方法（local method）
   - 对当前用户查询进行即时分析。
   - 主要方法：相关反馈（relevance feedback）。

2. 全局方法（global method）
   - 对整个文档集或外部资源做一次性全局分析。
   - 主要方法：查询扩展（query expansion）。

相关反馈：

1. 用户提交一个简短查询。
2. 搜索引擎返回一组文档。
3. 用户标记部分文档相关、部分文档不相关。
4. 系统根据标记结果计算新的查询表示。
5. 系统用新查询再次检索。
6. 理想情况下，新结果有更高召回率。

相关反馈分类：

- 显式相关反馈：用户明确标记相关或不相关。
- 隐式相关反馈：系统根据用户行为推测相关性，例如点击、停留时间。
- 伪相关反馈 / 盲反馈：无用户参与，系统假设初始返回的前 k 篇文档相关，并据此扩展查询。

查询扩展：

- 在查询中加入同义词或相关词项，提高召回率。
- 主要依赖同义词、近义词、查询日志等资源。

查询扩展类型：

- 人工构建的同义词/近义词词典，例如专业领域词表。
- 自动导出的同义词/近义词词典，例如基于词语共现统计。
- 基于查询日志挖掘出的查询等价类。

优点：

- 通常能提高召回率。
- 对专业搜索、科学工程领域搜索特别有用。

风险：

- 可能降低正确率。
- 对有歧义的词尤其危险，例如 `jaguar` 可指动物也可指汽车品牌。
- 人工词典维护成本高。

## 5. 上下文相似度

### 5.1 XML 检索中的上下文相似度

来源：`lecture10-xmlretrieval(1).pdf` pp.32-38

课件中“上下文相似度”出现在 XML 检索部分，英文为 Context resemblance。

XML 检索与普通非结构化检索的区别：

- 普通向量空间模型中，每一维通常对应一个词项。
- XML 检索中，文档具有结构，词项所在路径也有意义。
- 因此需要同时考虑词项和它在 XML 树中的位置。

词汇化子树（lexicalized subtree）：

- 把 XML 文档映射成包含词项和结构路径的子树。
- 查询和文档都表示到这些词汇化子树构成的向量空间中。
- 再使用向量相似度计算匹配程度。

结构化词项（structural term）：

```text
<c, t>
```

其中：

- `c` 是 XML 上下文或路径。
- `t` 是词项。

如果每一维只用普通词项，会退化为标准向量空间模型，无法区分结构。例如 `title` 中的 Gates 与 `author` 中的 Gates。  
如果每棵词汇化子树都作为一维，维度会过大。  
折中方案：对所有最终以单个词项结束的路径建立索引，即对 XML 上下文/词项对建立索引。

上下文相似度函数 `C_R`：

- 用来度量查询路径 `c_q` 和文档路径 `c_d` 的相似程度。
- `|c_q|` 和 `|c_d|` 分别表示查询路径和文档路径中的节点数。
- 当且仅当可以通过插入额外节点把 `c_q` 转换成 `c_d` 时，二者可以匹配。
- 完全相等时，`C_R(c_q, c_d) = 1.0`。

课件例子：

```text
C_R(c_q4, c_d2) = 3 / 4 = 0.75
C_R(c_q4, c_d3) = 3 / 5 = 0.6
```

可答要点：

- 上下文相似度衡量 XML 查询路径和文档路径的结构匹配程度。
- 路径越接近，相似度越高。
- 完全相同为 1。
- 可通过插入节点匹配的路径才可比较。

### 5.2 广告中的上下文匹配

来源：`lecture11-websearch(1).pdf` pp.16-18, p.29

如果考试中的“上下文相似度”放在广告部分，也可能指网页内容触发广告的上下文匹配。

互联网文本广告包括：

- 搜索关键词触发的广告，也称搜索广告。
- 网页内容触发的广告，也称上下文广告或内容匹配广告。

上下文广告的核心：

- 根据网页正文内容判断页面主题。
- 将广告内容与页面主题匹配。
- 广告与页面上下文越相关，越可能得到展示或点击。

排序时不能只看出价，还要看相关性。课件指出广告排序应结合投标价格和相关性，点击率 CTR 是相关度的重要指标。

## 6. Web 广告与次高价拍卖

### 6.1 传统广告的不足

来源：`lecture11-websearch(1).pdf` pp.11-14

传统广告包括：

- 品牌广告。
- 直接营销。

传统广告的不足：

- 广告投放场地或媒介有限，例如报纸、电视、杂志、橱窗、公交、电梯等。
- 广告场地费用高。
- 很难个性化。
- 投放效果很依赖广告商经验。
- 投放效果难以量化。

互联网广告的优点：

- 机会更多。
- 创意空间更大。
- 可以个性化处理。
- 每次点击成本较低。
- 可量化度更高。

### 6.2 互联网广告的主要形式

来源：`lecture11-websearch(1).pdf` pp.15-18

互联网广告主要形式：

1. 图片广告
   - 以图片、横幅等形式展示。

2. 文本广告
   - 搜索关键词触发广告。
   - 网页内容触发广告。

互联网文本广告的两类：

1. 搜索广告
   - Sponsored Search driven Ad / Paid Ad。
   - 由搜索关键词触发。
   - 例如 Google Adwords。

2. 网页广告 / 内容匹配广告 / 上下文广告
   - Web Page driven Ad / Contextual Ad。
   - 由网页内容触发。
   - 根据网页上下文选择广告。

### 6.3 搜索广告排序

来源：`lecture11-websearch(1).pdf` pp.20-31

第一代搜索广告：Goto 竞价排名。

- 广告商对关键词出价。
- 按投标价格排序。
- 用户点击后广告商付费。
- 问题：不区分广告和自然文档结果，没有相关度排序。

第二代搜索广告：Google。

- 严格区分自然搜索结果和搜索广告。
- 广告排序不只看 bid，还考虑相关性。

为什么不能只按出价排序：

- 可能展示与查询无关的广告。
- 用户体验差。
- 无关广告点击率低。
- 长期会损害搜索引擎整体收益。

替代方法：

```text
ad rank = bid * CTR
```

其中：

- `bid`：广告商愿意为每次点击支付的最高价格。
- `CTR`：clickthrough rate，点击率，等于 `clicks / impressions`。
- `ad rank`：广告排名分数，平衡出价和相关度。

其他排序因素还包括：

- 广告位置。
- 一天中的时间。
- 着陆页质量。
- 着陆页加载速度。
- 查询本身。

### 6.4 次高价拍卖

来源：`lecture11-websearch(1).pdf` pp.26-32；`lecture12-crawling(1).pdf` pp.5-6

次高价拍卖（second price auction）的基本原则：

```text
出价最高者获得物品，但支付第二高出价。
```

课件解释：

- 竞买人密封提交真实出价。
- 拍卖开始后打开报价。
- 最高出价者成交。
- 实际支付价格是第二高价。

为什么鼓励真实出价：

- 出价影响能否赢得拍卖。
- 一旦赢得拍卖，实际支付价格由第二高出价决定，而不是自己出价。
- 因此竞买人更有动力报出自己的真实心理价位。

搜索广告中的推广形式：广义次高价拍卖（GSP）思想。

Google 课件机制：

```text
ad rank = bid * CTR
```

广告商支付“维持当前排名所必须的价格”，通常由下一名广告的 `bid * CTR` 决定。

若广告商 1 排在广告商 2 前面，则广告商 1 的支付价格满足：

```text
paid_1 * CTR_1 = bid_2 * CTR_2
paid_1 = bid_2 * CTR_2 / CTR_1
```

课件例子：

```text
p1 = 3.00 * 0.03 / 0.06 = 1.50
p2 = 1.00 * 0.08 / 0.03 = 2.67
p3 = 4.00 * 0.01 / 0.08 = 0.50
```

考试可答：

- 普通次高价拍卖：最高出价者胜出，支付第二高价。
- 搜索广告中还要结合 CTR，因为广告位价值与点击概率有关。
- 排名使用 `bid * CTR`，支付价格由下一名的排序分数除以自己的 CTR 得到。

## 7. 高频考试问答模板

### 7.1 问：为什么倒排记录表合并是 O(x+y)？

答：两个 postings list 都按 docID 升序排列，可以用双指针从头到尾扫描。每次比较当前 docID，较小者指针后移，相等则输出并两个指针都后移。每个元素最多访问一次，所以时间复杂度是两个表长之和 `O(x+y)`。

### 7.2 问：布尔查询如何优化？

答：对多个 `AND` 条件，优先处理 `df` 最小、postings list 最短的词项，使中间结果尽早变小。对含 `OR` 的一般表达式，可以用 OR 子表达式中各词项 `df` 之和估计结果大小，再按估计大小从小到大处理。

### 7.3 问：MapReduce 如何构建倒排索引？

答：Map 阶段读文档并输出 `<term, docID>`；Shuffle 阶段按 term 分组；Reduce 阶段收集同一 term 的全部 docID，排序、合并并写成 postings list。大规模场景下由 Master 分配 split 给 Parser，再由 Inverter 负责生成倒排记录表。

### 7.4 问：为什么 IR 中要压缩索引？

答：压缩可以减少磁盘空间，让更多索引进入内存，降低磁盘 I/O，加快从磁盘到内存的数据传输。只要解压足够快，读取压缩数据再解压通常比直接读取未压缩数据更快。

### 7.5 问：Jaccard、tf、idf、tf-idf 分别解决什么问题？

答：Jaccard 衡量两个集合的重合度，但不考虑词频和词项稀有性。对数 tf 处理词项在文档中出现越多越重要但不线性增长的问题。IDF 处理罕见词信息量更高的问题。TF-IDF 把文档内词频和全局稀有性相乘，是经典权重计算方法。

### 7.6 问：Precision 和 Recall 的区别？

答：Precision 衡量返回结果中有多少是真的相关，公式是相关且返回 / 全部返回。Recall 衡量全部相关文档中有多少被返回，公式是相关且返回 / 全部相关。Precision 重在“准”，Recall 重在“全”。

### 7.7 问：搜索结果摘要有哪些类型？

答：两类：静态摘要和动态摘要。静态摘要与查询无关，常取文档前若干词或重要句子。动态摘要依赖查询，选取包含查询词项的窗口 snippet，优先显示包含完整短语或查询词密集出现的位置。

### 7.8 问：如何提高召回率？

答：主要通过相关反馈和查询扩展。相关反馈根据用户标注的相关/不相关文档重构查询；查询扩展通过加入同义词、近义词或查询日志中相关表达来返回更多相关文档。它们通常能提高召回率，但可能降低正确率。

### 7.9 问：什么是上下文相似度？

答：在 XML 检索中，上下文相似度衡量查询路径和文档路径的结构相似程度。词项不仅看内容，还看它出现在 XML 树中的路径。路径完全相同时相似度为 1；路径越接近，相似度越高。

### 7.10 问：次高价拍卖为什么适合搜索广告？

答：普通次高价拍卖中，最高出价者获胜但支付第二高价，这鼓励真实出价。搜索广告中还要考虑广告相关性，所以用 `ad rank = bid * CTR` 排序，并按维持当前排名所需的价格付费，即下一名排序分数除以自己的 CTR。

```text
极简倒排索引实现
def build_inverted_index(docs):
index = {}
for doc_id, text in docs:
words = text.split()
遍历单词，记录每个词在当前文档的位置
word_pos = {}
for pos, word in enumerate(words):
if word not in word_pos:
word_pos[word] = []
word_pos[word].append(pos)
写入倒排索引
for word, pos_list in word_pos.items():
count = len(pos_list)
entry = (doc_id, count, pos_list)
if word not in index:
index[word] = []
index[word].append(entry)
return index
```
